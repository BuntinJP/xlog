## dehydratedからacme.shへ

以前、dehydratedを使って証明書を自動更新していました。

[Cloudflareでドメインを管理するブログのSSL証明書をより強固に自動更新したい[dehydrated]](/posts/welcome-dehydrated)

しかし、一つ不満な点がありました。それは、CloudflareのAPIを叩くためのスクリプトが、dehydrated本体とは別のプラグインだった点です。  
動き出してしまえば、更新などおそらく不要ではあるのですが、Let'sEncryptや、Cloudflareの権限モデルの変化など、将来的になにか変更があったときなどに、どれくらいのサポートが行われるのか不明瞭です。

また、利用しているプラグインが古く、新しいCloudflareの認証形式のAPI Tokenに対応していませんでした。

過去記事では、dehydratedをcertbotと比較して、以下の点を評価していました。

- bashスクリプトであること
- 余計な機能がないこと

これを満たしつつ、acme.shは更に以下の点で優れています。

- CloudflareAPIコネクタが公式で実装されている
- bashのみならず、unixシェル(dash,sh)をサポートしている
- パッケージマネージャ経由でのインストールに対応している(FreeBSDのpkg,ArchLinuxのpacman!)

公式でCloudflareサポートはアツいです。何かあっても、`paru -Syu acme.sh`して再実行すればいい...  
なんと素晴らしいことでしょう！

堅牢さでbashスクリプトであることを偉そうに語ってましたが、お恥ずかしいです。  
dash, sh は強すぎます笑

ArchLinuxで使う以上、bashサポートさえあれば動きます。しかし、同じ対応をUbuntu(debian系はdash)でもやりたくなったときを考えると、より汎用性が高いのはacme.shと言えます。

また、dehydratedは証明書を新しく発行すると、リンクを張り替えるという挙動をするのですが、正直期限が切れた証明書を保持する理由はないので、その点の機能低下は評価に反映していません。

ちなみにacme.shは、前記事にてこき下ろしたZeroSSLがスポンサーをしています笑

「実装自体は巨大化してないか?」という批判はごもっともです。しかし、Cloudflareサポートは私にとっては不要な機能ではなく、むしろ必須の機能です。  
これが公式で実装されているacme.shとdehydratedでは、やはりacme.shはよりよい選択肢と言わざるを得ません。

ということで、acme.shをArchLinuxに導入します。

## 手順

acme.shを、公式パッケージからインストールしてセットアップする手順です。  
また、Nginxをリロードする設定も導入します。

### 注意

- root前提です
- 更新スクリプト内にAPIトークンを保存します
- `/etc/ssl/`以下に証明書を保存します

### 2. APIトークン・アカウントIDを取得

- 発行するドメインをゾーンに持つCloudflareのAPIトークンを取得

Cloudflareのダッシュボードから、APIトークンを取得します。

https://dash.cloudflare.com/profile/api-tokens からAPIトークンを取得します。

![14-acme.sh/01-cf-api-token-setting.webp](/images/14-acme.sh/01-cf-api-token-setting.webp)

画像のように権限設定します。  
隠れているのは、実行するドメインと、IPアドレスです。

注意点として、IP制限する場合、IPv4,IPv6の区別があります。

得られたAPIトークンは、以下のコマンドで検証できます。

```console
># curl -fsS "https://api.cloudflare.com/client/v4/user/tokens/verify" \
-H "Authorization: Bearer <API_TOKEN>" \
| jq -r '.result.status'
---
active
```

- アカウントIDを取得

ダッシュボードから確認できます。

隠す場所多くて面倒なのと簡単に見つけられるので公式ドキュメントで失礼します。

https://developers.cloudflare.com/fundamentals/account/find-account-and-zone-ids/

### 4. Webサーバーを設定

証明書配置予定のパスに、事前に設定を変更しておきます。

当記事での配置先
キー: `/etc/ssl/private/<DOMAIN>/privkey.pem`
フルチェーン: `/etc/ssl/certs/<DOMAIN>/fullchain.pem`

```console
># vim /etc/nginx/conf.d/01_<DOMAIN>.conf # 例です
># grep "ssl_certificate" /etc/nginx/conf.d/01_<DOMAIN>.conf
    ssl_certificate     /etc/ssl/certs/<DOMAIN>/fullchain.pem;
    ssl_certificate_key /etc/ssl/private/<DOMAIN>/privkey.pem;
```

`nginx -t && systemctl reload nginx`
を、証明書更新時に実行するように設定するため、事前に成功することを確認しておく必要があります。

### 5. 証明書配置

acme.shの機能を使い、証明書を設置します。

```console
># acme.sh --install-cert -d "<DOMAIN>" \
  --key-file "/etc/ssl/private/<DOMAIN>/privkey.pem" \
  --fullchain-file "/etc/ssl/certs/<DOMAIN>/fullchain.pem" \
  --reloadcmd "nginx -t && systemctl reload nginx"
[Mon Mar 30 12:56:16 AM JST 2026] The domain '<DOMAIN>' seems to already have an ECC cert, let's use it.
[Mon Mar 30 12:56:16 AM JST 2026] Installing key to: /etc/ssl/private/<DOMAIN>/privkey.pem
[Mon Mar 30 12:56:16 AM JST 2026] Installing full chain to: /etc/ssl/certs/<DOMAIN>/fullchain.pem
[Mon Mar 30 12:56:16 AM JST 2026] Running reload cmd: nginx -t && systemctl reload nginx
nginx: the configuration file /etc/nginx/nginx.conf syntax is ok
nginx: configuration file /etc/nginx/nginx.conf test is successful
[Mon Mar 30 12:56:16 AM JST 2026] Reload successful
```

`--reloadcmd`で指定したコマンドが、証明書更新時に実行される内容です。

### 6. 自動更新設定

スクリプトを作成し、cronに登録します。

```console
># vim /usr/local/bin/renew-cert.sh # 例です
># chmod +x /usr/local/bin/renew-cert.sh
># crontab -e
># crontab -l | grep renew-cert.sh
0 0 * * 0 /usr/local/bin/renew-cert.sh 2>&1
># # 毎週日曜日 0時0分
```

```bash title="renew-cert.sh"
#!/bin/sh
set -eu

DOMAIN="<DOMAIN>"

CF_Token='<API_TOKEN>'
CF_Account_ID='<ACCOUNT_ID>'

export CF_Token
export CF_Account_ID

/usr/bin/acme.sh --server letsencrypt --renew -d "$DOMAIN"
```

心配であれば、`--force`を付けて期限を無視して再発行できます。

以上で更新設定まで完了です。
APIトークンの期限が切れたりしない限り稼働し続けます。

## まとめ

この記事では、acme.shを使い、Nginxで公開しているWebサイトの証明書をLet's Encryptで取得し、自動更新する設定を行いました。  
dehydratedより強固な証明書発行プロセスを導入できたかと思います。
